    39→1. Pengenalan & Latar Belakang
    40→ADTEC Kota Tinggi (ADTEC KT) merupakan sebuah institusi latihan teknikal di bawah Jabatan Tenaga Manusia (JTM), Kementerian Sumber
    41→Manusia. Sebagai sebahagian daripada tanggungjawab akademik dan kokurikulum, para pengajar di setiap bengkel (workshop) kerap
    42→menganjurkan pelbagai aktiviti pelajar – termasuk lawatan industri, program khidmat masyarakat, pertandingan kemahiran, seminar, dan
    43→program kerjasama industri.
    44→Pada masa ini, rekod aktiviti-aktiviti tersebut dikumpul dan disahkan secara manual (borang kertas / emel / Excel yang dikongsi),
    45→menyebabkan proses semakan oleh Penyelaras dan pengesahan oleh Penolong Pengarah mengambil masa yang lama, sukar dijejaki, dan
    46→tiada paparan statistik pelaksanaan secara masa nyata (real-time) mengikut daerah, negeri, atau tempoh masa.
    47→Sistem Rekod Aktiviti Pelajar ADTEC KT dicadangkan untuk menggantikan proses manual ini dengan aplikasi web berpusat yang
    48→membolehkan aliran kerja tiga peringkat – input oleh Pengajar → semakan oleh Penyelaras → pengesahan oleh Penolong
    49→Pengarah – berlaku secara digital, telus, dan boleh dijejaki, lengkap dengan papan pemuka statistik automatik.
    50→2. Objektif Sistem
    51→Menyediakan platform digital berpusat untuk merekod aktiviti pelajar ADTEC KT menggantikan proses manual berasaskan kertas/Excel.
    52→Mewujudkan aliran kerja kelulusan tiga peringkat (Pengajar → Penyelaras → Penolong Pengarah) yang jelas, telus dan boleh diaudit.
    53→Membolehkan pengajar menghantar data aktiviti secara individu atau pukal (bulk) untuk semakan.
    54→Membolehkan Penolong Pengarah mengesahkan atau menolak data secara individu atau pukal.
    55→Menjana statistik pelaksanaan aktiviti secara automatik mengikut tempoh (harian, mingguan, bulanan, tahunan) dan mengikut lokasi
    56→(daerah & negeri).
    57→Memaparkan statistik ringkas pelaksanaan aktiviti yang telah disahkan pada halaman utama (dashboard awam dalaman) sistem.
    58→Membina sistem menggunakan seni bina moden: backend/pangkalan data Supabase, antara muka dijana melalui z.ai (model GLM 5.2),
    59→dan deployment melalui Netlify, dengan reka bentuk UI/UX bergaya glassmorphism.
    60→3. Skop Projek

    91→5. Peranan & Pengguna Sistem
    92→Sistem menggunakan model akses berasaskan peranan (Role-Based Access Control – RBAC) dengan log masuk (username/e-mel & kata
    93→laluan) wajib bagi setiap peranan.
    94→1. Pengajar (Instructor)
    95→Log masuk menggunakan akaun sendiri (dikaitkan dengan Bengkel tertentu).
    96→Memasukkan (create) rekod aktiviti baharu.
    97→Melihat senarai aktiviti yang telah dimasukkan (status Draf) sebelum dihantar.
    98→Memadam (delete) rekod satu persatu jika terdapat kesilapan sebelum dihantar.
    99→Menghantar rekod kepada Penyelaras – secara individu (satu-satu) atau pukal (bulk submit, pilih banyak rekod sekali gus).
   100→Melihat status rekod yang telah dihantar (Menunggu Semakan / Disemak / Disahkan / Ditolak) – termasuk sebab penolakan jika
   101→ada.
   102→2. Penyelaras (Coordinator)
   103→Log masuk dan melihat semua rekod berstatus "Menunggu Semakan" daripada semua Pengajar di bawah seliaannya.
   104→Menyemak (review) butiran setiap rekod aktiviti.
   105→Menghantar (forward) rekod yang telah disemak kepada Penolong Pengarah untuk pengesahan – individu atau pukal.
   106→Boleh menghantar semula (kembalikan) rekod kepada Pengajar berkenaan jika terdapat maklumat tidak lengkap/tidak tepat,
   107→berserta catatan.
   108→Melihat statistik ringkas rekod yang telah/belum disemak.
   109→3. Penolong Pengarah (Assistant Director)
   110→Log masuk dan melihat semua rekod berstatus "Menunggu Pengesahan" yang telah dihantar oleh Penyelaras.
   111→Mengesahkan (approve) rekod secara individu atau pukal.
   112→Menolak (reject) rekod secara individu, berserta catatan/sebab penolakan yang wajib diisi.
   113→Mengakses papan pemuka statistik penuh (harian/mingguan/bulanan/tahunan, mengikut daerah/negeri) bagi semua rekod yang
   114→telah disahkan.
   115→4. Admin (Pilihan – disyorkan untuk urus tetap sistem)
   116→Mengurus akaun pengguna (cipta/kemaskini/nyahaktif akaun Pengajar, Penyelaras, Penolong Pengarah).
   117→Mengurus data rujukan (senarai Bengkel, Negeri, Daerah).
   118→Melihat log audit keseluruhan sistem.

   121→6. Aliran Kerja Keseluruhan Sistem (Workflow)
   122→Setiap rekod aktiviti melalui lima (5) status sepanjang kitaran hayatnya:
   123→1. DRAF 2. MENUNGGU SEMAKAN 3. MENUNGGU PENGESAHAN 4a. DISAHKAN
   124→Pengajar input data, boleh ➤ Dihantar oleh Pengajar kepada ➤ Dihantar oleh Penyelaras kepada ➤ Diluluskan oleh Penolong Pengarah
   125→edit/padam Penyelaras Penolong Pengarah – masuk statistik
   126→4b. DITOLAK (oleh Penyelaras) 4c. DITOLAK (oleh Penolong Pengarah)
   127→Kembali ke Pengajar untuk pembetulan → kembali ke Draf Kembali ke Penyelaras / Pengajar berserta sebab → kembali ke Draf
   128→Nota: Rekod yang "Ditolak" pada mana-mana peringkat akan dikembalikan ke status Draf pada senarai Pengajar berkenaan, lengkap dengan catatan sebab penolakan, supaya
   129→boleh dibetulkan dan dihantar semula.
   130→6.1 Gambar Rajah Aliran Ringkas (Swimlane)
   131→Peringkat Tindakan Status Terhasil
   132→Pengajar Cipta rekod → simpan sebagai draf → semak senarai sendiri → padam DRAF → MENUNGGU_SEMAKAN
   133→jika salah → hantar (individu/pukal)
   134→Penyelaras Semak senarai masuk → buka butiran → hantar ke PP (individu/pukal) MENUNGGU_SEMAKAN →
   135→atau pulangkan ke Pengajar MENUNGGU_PENGESAHAN atau → DRAF (ditolak)
   136→Penolong Pengarah Semak senarai masuk → sahkan (individu/pukal) atau tolak (individu, MENUNGGU_PENGESAHAN → DISAHKAN atau →
   137→dengan sebab) DRAF (ditolak)
   138→Sistem Kemaskini statistik automatik apabila status = DISAHKAN Data masuk paparan Statistik & Dashboard

   142→7.1 Modul Log Masuk & Pengesahan Identiti
   143→FR-1.1 Sistem mesti menyediakan skrin log masuk (e-mel/ID pengguna + kata laluan) untuk keempat-empat peranan.

   145→pemuka mengikut peranan selepas log masuk berjaya.
   146→FR-1.3 Sistem mesti memaparkan mesej ralat sekiranya kata laluan/ID salah, dan menyekat sesi tanpa log masuk yang sah (route
   147→protection).
   148→FR-1.4 Sistem mesti menyediakan fungsi "Log Keluar" dan "Tukar Kata Laluan".
   149→FR-1.5 Sistem mesti menyediakan fungsi "Lupa Kata Laluan" (reset melalui e-mel, disokong oleh Supabase Auth).
   150→7.2 Modul Pengajar
   151→FR-2.1 Pengajar boleh memasukkan rekod aktiviti baharu melalui borang dengan medan: tarikh pelaksanaan, nama aktiviti, bengkel
   152→terlibat, negeri, daerah, dan nama pengurus aktiviti.
   153→FR-2.2 Sistem mesti memilih senarai Daerah secara dinamik berdasarkan Negeri yang dipilih (dropdown berantai).
   154→FR-2.3 Pengajar boleh melihat senarai semua rekod yang telah dimasukkan (status DRAF) dalam bentuk jadual sebelum dihantar.
   155→FR-2.4 Pengajar boleh mengedit rekod berstatus DRAF sebelum dihantar.
   156→FR-2.5 Pengajar boleh memadam (delete) rekod satu persatu daripada senarai draf jika terdapat kesilapan.
   157→FR-2.6 Pengajar boleh menghantar rekod kepada Penyelaras secara individu (satu rekod) melalui butang "Hantar" pada setiap baris.
   158→FR-2.7 Pengajar boleh memilih pelbagai rekod (checkbox) dan menghantar secara pukal (bulk submit) melalui satu butang "Hantar
   159→Terpilih".
   160→FR-2.8 Pengajar boleh melihat senarai berasingan bagi rekod mengikut status: Draf, Menunggu Semakan, Menunggu Pengesahan,
   161→Disahkan, Ditolak – lengkap dengan sebab penolakan (jika berkaitan).
   162→7.3 Modul Penyelaras
   163→FR-3.1 Penyelaras boleh melihat senarai semua rekod berstatus "Menunggu Semakan" dalam bentuk jadual.
   164→FR-3.2 Penyelaras boleh membuka butiran penuh setiap rekod untuk semakan.
   165→FR-3.3 Penyelaras boleh menghantar rekod yang disemak kepada Penolong Pengarah secara individu atau pukal (checkbox multi-select).
   166→FR-3.4 Penyelaras boleh memulangkan (reject) rekod kepada Pengajar berkenaan berserta catatan ringkas sebab pemulangan.
   167→FR-3.5 Sistem mesti menghantar notifikasi dalam-sistem (in-app) kepada Pengajar apabila rekod dipulangkan.
   168→7.4 Modul Penolong Pengarah
   169→FR-4.1 Penolong Pengarah boleh melihat senarai semua rekod berstatus "Menunggu Pengesahan" dalam bentuk jadual.
   170→FR-4.2 Penolong Pengarah boleh mengesahkan (approve) rekod secara individu melalui butang pada setiap baris.
   171→FR-4.3 Penolong Pengarah boleh memilih pelbagai rekod dan mengesahkan secara pukal (bulk approve).
   172→FR-4.4 Penolong Pengarah boleh menolak (reject) rekod secara individu, dengan medan catatan sebab penolakan yang wajib diisi.

   222→10. Reka Bentuk UI/UX – Glassmorphism
   223→Reka bentuk visual sistem menggunakan gaya Glassmorphism – kad/panel separa lutsinar dengan kesan kabur latar (backdrop-filter:
   224→blur()), sempadan nipis bercahaya, bayang lembut, dan latar belakang gradien berwarna supaya kesan "kaca" kelihatan jelas.
   225→10.1 Prinsip Reka Bentuk
   226→Latar belakang aplikasi menggunakan gradien lembut (contoh: biru laut → teal → putih) supaya lapisan kaca kelihatan jelas di atasnya.
   227→Semua kad (dashboard cards, borang, jadual, modal) menggunakan background: rgba(255,255,255,0.15–0.25), backdrop-filter:
   228→blur(12–20px), sempadan 1px solid rgba(255,255,255,0.3), dan border-radius: 16–20px.
   229→Bayang lembut (box-shadow) digunakan untuk kedalaman (depth), bukan garis keras.
   230→Butang utama menggunakan gradien warna jenama (teal/biru) dengan kesan "hover glow".
   231→Ikon menggunakan set garis nipis (outline icon set) – konsisten dan ringan.
   232→Tipografi: fon sans-serif moden (contoh: Inter / Poppins / Segoe UI), hierarki jelas antara tajuk, subtajuk dan teks badan.
   233→Mod terang (light mode) sebagai lalai; mod gelap (dark mode) glassmorphism disediakan sebagai pilihan pengguna (togol).
   234→10.2 Palet Warna Dicadangkan
   235→Peranan Warna Kod Hex Kegunaan
   236→Primary – Deep Teal #0D3B52 Tajuk, teks penting, latar navigasi

   361→13. Row Level Security (RLS) & Keselamatan
   362→RLS diaktifkan pada semua jadual data (aktiviti, profiles, audit_log). Polisi umum adalah seperti berikut:
   363→Peranan SELECT INSERT UPDATE
   364→Pengajar Hanya rekod yang dicipta sendiri (semua Boleh cipta rekod baharu (status = Hanya rekod sendiri berstatus 'draf'
   365→status, untuk jejak status) draf, dicipta_oleh = auth.uid()) (edit/padam/hantar)
   366→Penyelaras Rekod berstatus 'menunggu_semakan' & ke Tiada (tidak cipta rekod) Rekod berstatus 'menunggu_semakan'
   367→atas dalam bengkel seliaannya sahaja (hantar ke PP / pulangkan)
   368→Penolong Pengarah Rekod berstatus 'menunggu_pengesahan' & Tiada Rekod berstatus
   369→'disahkan' (semua bengkel/daerah) 'menunggu_pengesahan' sahaja
   370→(sah/tolak)
   371→Admin Semua rekod Semua Semua (urus tetap)
   372→13.1 Contoh Polisi RLS (SQL)
   373→alter table aktiviti enable row level security;
   374→-- Pengajar hanya nampak rekod sendiri
   375→create policy "pengajar_select_own"
   376→on aktiviti for select
   377→using (
   378→dicipta_oleh = auth.uid()
   379→or exists (
   380→select 1 from profiles p
   381→where p.id = auth.uid()
   382→and p.role in ('penyelaras','penolong_pengarah','admin')
   383→)
   384→);
   385→-- Pengajar hanya boleh insert rekod sendiri berstatus draf
   386→create policy "pengajar_insert_own"
   387→on aktiviti for insert
   388→with check (dicipta_oleh = auth.uid());
   389→-- Pengajar hanya boleh update/padam rekod sendiri semasa draf
   390→create policy "pengajar_update_own_draf"
   391→on aktiviti for update
   392→using (dicipta_oleh = auth.uid() and status = 'draf');
   393→-- Penyelaras hanya boleh update rekod status menunggu_semakan
   394→create policy "penyelaras_update_review"
   395→on aktiviti for update
   396→using (
   397→status = 'menunggu_semakan'
   398→and exists (select 1 from profiles p where p.id = auth.uid() and p.role = 'penyelaras')
   399→);
   400→-- Penolong Pengarah hanya boleh update rekod status menunggu_pengesahan
   401→create policy "pp_update_confirm"
   402→on aktiviti for update
   403→using (
   404→status = 'menunggu_pengesahan'
   405→and exists (select 1 from profiles p where p.id = auth.uid() and p.role = 'penolong_pengarah')
   406→);
   407→13.2 Keselamatan Tambahan
   408→Semua kata laluan diuruskan & di-hash oleh Supabase Auth (bcrypt) – tiada kata laluan disimpan dalam bentuk teks biasa.
   409→Sesi log masuk menggunakan JSON Web Token (JWT) dengan tempoh luput (expiry) & refresh token.
   410→Kunci service_role Supabase TIDAK dibenarkan digunakan pada sisi frontend; hanya anon key + RLS digunakan pada client.
   411→Edge Function berasingan (menggunakan service role secara selamat di sisi server) digunakan untuk operasi pukal (bulk
   412→approve/submit) supaya konsisten dari segi transaksi.
   413→Log audit tidak boleh dipadam oleh mana-mana peranan selain Admin (append-only bagi Pengajar/Penyelaras/PP).

   475→15. Modul Statistik & Papan Pemuka
   476→Statistik dikira hanya berdasarkan rekod berstatus disahkan, menggunakan tarikh_pelaksanaan sebagai asas pengiraan tempoh.
   477→15.1 Statistik Mengikut Tempoh
   478→Tempoh Kaedah Pengiraan
   479→Harian Kumpulan (group by) mengikut tarikh_pelaksanaan; papar bilangan aktiviti bagi 30 hari terkini (line/bar chart).
   480→Mingguan Kumpulan mengikut minggu ISO (Isnin–Ahad); papar bilangan bagi 12 minggu terkini.
   481→Bulanan Kumpulan mengikut bulan/tahun; papar bilangan bagi 12 bulan terkini.
   482→Tahunan Kumpulan mengikut tahun; papar trend beberapa tahun (jika data tersedia).
   483→15.2 Statistik Mengikut Lokasi
   484→Bilangan pelaksanaan aktiviti mengikut Daerah – boleh ditapis mengikut tempoh (harian/mingguan/bulanan/tahunan).
   485→Bilangan pelaksanaan aktiviti mengikut Negeri – boleh ditapis mengikut tempoh yang sama.
   486→Paparan carta bar/peta ringkas bagi perbandingan antara negeri/daerah.
   487→15.3 Halaman Utama (Dashboard Awam Dalaman)
   488→Selepas log masuk, halaman utama memaparkan kad ringkasan glassmorphism yang menunjukkan:
   489→Jumlah keseluruhan aktiviti disahkan (kesemua masa).
   490→Jumlah aktiviti disahkan bulan semasa vs bulan lepas (peratus perubahan).
   491→Carta ringkas 6 bulan terkini (trend bulanan).
   492→3 Negeri/Daerah paling aktif (top locations) bagi tempoh semasa.
   493→(Khusus Pengajar/Penyelaras) – bilangan rekod menunggu tindakan masing-masing.
   494→15.4 Contoh Query SQL Statistik Bulanan
   495→select
   496→to_char(date_trunc('month', tarikh_pelaksanaan), 'YYYY-MM') as bulan,
   497→count(*) as jumlah_aktiviti
   498→from aktiviti
   499→where status = 'disahkan'
   500→group by 1
   501→order by 1;
   502→select n.nama_negeri, count(*) as jumlah_aktiviti
   503→from aktiviti a
   504→join negeri n on n.id = a.negeri_id
   505→where a.status = 'disahkan'
   506→and a.tarikh_pelaksanaan >= date_trunc('year', current_date)
   507→group by n.nama_negeri
   508→order by jumlah_aktiviti desc;

   601→-- 3. PROFIL PENGGUNA (dipautkan ke auth.users)
   602→-- =====================================================
   603→create table profiles (
   604→id uuid primary key references auth.users(id) on delete cascade,
   605→full_name text not null,
   606→email text not null,
   607→role user_role not null,
   608→bengkel_id uuid references bengkel(id),
   609→created_at timestamptz default now()
   610→);
   611→-- =====================================================
   612→-- 4. JADUAL AKTIVITI (UTAMA)
   613→-- =====================================================
   614→create table aktiviti (
   615→id uuid primary key default gen_random_uuid(),
   616→tarikh_pelaksanaan date not null,
   617→nama_aktiviti text not null,
   618→bengkel_id uuid references bengkel(id) not null,
   619→negeri_id uuid references negeri(id) not null,
   620→daerah_id uuid references daerah(id) not null,
   621→nama_pengurus_aktiviti text not null,
   622→status aktiviti_status not null default 'draf',
   623→catatan_penolakan text,
   624→dicipta_oleh uuid references profiles(id) not null,
   625→disemak_oleh uuid references profiles(id),
   626→disahkan_oleh uuid references profiles(id),
   627→dihantar_semakan_pada timestamptz,
   628→dihantar_pengesahan_pada timestamptz,
   629→disahkan_pada timestamptz,
   630→created_at timestamptz default now(),
   631→updated_at timestamptz default now()
   632→);
   633→-- =====================================================
   634→-- 5. LOG AUDIT
   635→-- =====================================================
   636→create table audit_log (
   637→id bigint generated always as identity primary key,
   638→aktiviti_id uuid references aktiviti(id) not null,
   639→tindakan text not null,
   640→oleh uuid references profiles(id) not null,
   641→catatan text,
   642→dicipta_pada timestamptz default now()
   643→);
   644→-- =====================================================
   645→-- 6. AKTIFKAN RLS
   646→-- =====================================================
   647→alter table aktiviti enable row level security;
   648→alter table profiles enable row level security;
   649→alter table audit_log enable row level security;
